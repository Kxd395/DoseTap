# DoseTap Update Brief (Aligned to Pain Detail Requirements)

## Why this update exists
The prior brief was too generic for pain capture. The app must support multiple pain regions at once, different sides per region, and different sensation types per entry.

Required real-world example:
- Mid back pain: `2/10`, both sides, throbbing.
- Lower back pain: `9/10`, right side only, sharp + shooting.

These cannot be merged into one pain value.

## Source of truth
Implementation must follow:
- `/Volumes/Developer/projects/DoseTap/docs/review/checkin_questionnaires_spec_2026-02-13.md`

This file is an execution summary and checklist.

## Non-negotiable model rules
1. Pain is stored as `pain.entries[]` (repeatable), not one combined location field.
2. Each entry must include:
- `area`
- `side`
- `intensity` (`0...10` integer)
- `sensations[]` (at least one)
3. Entry identity key is deterministic:
- `entry_key = "{area}|{side}"`
4. If left/right details differ, they must be separate entries.
- Allowed: `lower_back|left`, `lower_back|right`
- Not allowed: one `lower_back|both` with mixed values.

## Questionnaire behavior (Pre-night and Morning)
Both check-ins must ask the same core pain structure so deltas can be computed.

### Pain flow
1. Ask `pain.any`.
2. If yes, user selects one or more areas.
3. For each selected area, open detail sheet/popover.
4. User sets side, intensity, and sensations for that area+side.
5. Repeat until all selected areas are completed.

### Required UI output
- Completed chips/tokens per entry, for example:
  - `Mid Back (Both) 2/10 - Throbbing`
  - `Lower Back (Right) 9/10 - Sharp, Shooting`
- Submission blocked until required entry fields are complete.

## Morning vs Pre-night comparison
Comparison is by `entry_key` (`area|side`) only.

For each key in union(pre, morning):
- key in both: `delta = morning.intensity - pre.intensity`
- only in morning: `status = new`
- only in pre: `status = resolved`

Display examples:
- `mid_back|both: -1` improved
- `lower_back|right: +2` worsened
- `neck|left: new`
- `upper_back|both: resolved`

## Storage requirements
Each submission must persist:
- `id`
- `session_id`
- `session_date`
- `checkin_type` (`pre_night` or `morning`)
- `questionnaire_version`
- `submitted_at_utc`
- `local_offset_minutes`
- `responses_json` (contains full `pain.entries[]`)

Recommended indexed fields for analytics:
- `pain.entry.{entry_key}.intensity`
- `pain.entry.{entry_key}.sensations`

## Validation checklist
- `pain.any = true` requires `pain.entries.count >= 1`.
- Every entry has `area`, `side`, `intensity`, and `sensations[1+]`.
- No duplicate `entry_key` in one submission.
- `intensity` must be integer `0...10`.

## Test cases (must pass)
1. Save mixed pain scenario:
- `mid_back|both = 2, throbbing`
- `lower_back|right = 9, sharp+shooting`
- Verify two distinct entries are stored.

2. Side split enforcement:
- `lower_back left=4` and `lower_back right=8`
- Verify two entries, not `both`.

3. Morning delta rendering:
- Pre `mid_back|both=2`, Morning `mid_back|both=1`
- Show `-1`.

4. New/resolved entries:
- Pre has `upper_back|both`, Morning missing -> `resolved`
- Morning has `neck|left` only -> `new`

5. Version safety:
- Historical submissions remain readable after question updates.

## Scope reminder
This brief updates question logic and data mapping only. It does not replace timeline UX work, but its output must be compatible with timeline review, dashboard analytics, and future CloudKit sync.

## Added scope for substances (pre-night granularity)
Pre-night must also capture granular substance details:

- Caffeine:
  - source
  - last intake time
  - last amount (mg)
  - daily total (mg)
- Alcohol:
  - level
  - last drink time
  - last amount (drinks)
  - daily total (drinks)

Rules:
- If caffeine/alcohol is not `none`, all detail fields are required.
- Daily total must be `>=` last amount.
- These fields must be written to normalized `responses_json` for analytics and exports.

## Added scope for activity and naps (pre-night granularity)
Pre-night must also capture detailed activity and nap tracking:

- Exercise:
  - level
  - type
  - last exercise time
  - duration minutes
- Naps:
  - duration bucket
  - nap count
  - total nap minutes
  - last nap end time

Rules:
- If exercise is not `none`, all exercise detail fields are required.
- If nap is not `none`, all nap detail fields are required.
- These values must be written to normalized `responses_json` for downstream metrics and dashboards.
